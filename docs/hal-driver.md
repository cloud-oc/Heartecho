# HAL Driver Path

The visible virtual audio device is not created by SwiftUI. It is created by a Core Audio HAL Audio Server Driver plug-in that Core Audio loads into the audio server process.

## Bundle Shape

A production driver bundle needs:

- `Contents/Info.plist` with the audio server plug-in factory UUID.
- A binary exporting a factory function for the plug-in COM-style interface.
- Real implementations for object/property queries, stream format negotiation, device start/stop, and IO callbacks.
- A signed and notarized installer flow.

The repository can now build a local, unsigned `.driver` artifact with `scripts/build-hal-bundle.sh debug` and verify the bundle structure and factory export with `scripts/verify-hal-bundle.sh`. `scripts/sign-hal-bundle.sh` previews or runs `codesign`, `scripts/check-hal-signing.sh` reports current signing state, and `scripts/install-hal-bundle.sh` refuses `--execute` installs unless signature verification passes or `--allow-unsigned` is explicitly provided. This is still a packaging scaffold; real production installation requires Developer ID signing and notarization.

`HALDriverProbe` is the current app/diagnostic visibility check. It inspects the local build artifact, user and system HAL install locations, bundle metadata, code-signature validity, and Core Audio devices whose UIDs match the driver bundle prefix.

After installation, `scripts/reload-core-audio.sh` can preview or request a `coreaudiod` restart, and `swift run HeartechoDiagnostics --wait-hal-device 30` polls Core Audio until a matching virtual device appears or the timeout expires.

Typical install locations are:

- `/Library/Audio/Plug-Ins/HAL`
- `~/Library/Audio/Plug-Ins/HAL`

The exact location depends on distribution policy and whether the product needs all-user availability.

## Driver Responsibilities

- Publish each configured virtual device.
- Expose input/output streams and channel names.
- Accept render data from app/process/hardware sources.
- Provide zeroed audio when a source is missing.
- Report property changes when routes, names, sample rates, or channel counts change.
- Expose practical runtime device properties such as latency, safety offset, buffer frame size/range, and virtual transport type.
- Remain real-time safe in IO paths.

## App Responsibilities

- Edit and validate the rich routing graph.
- Discover hardware devices and application/process sources.
- Request permissions.
- Persist configuration.
- Notify the driver/helper when the graph changes.
- Present diagnostics when the driver is missing, unsigned, blocked, or stale.

## Suggested Bridge

Start with the Swift-side graph for prototyping, then move to a helper service:

1. App writes a validated graph to Application Support or an App Group container.
2. Helper validates the graph again and translates it to compact runtime config.
3. Driver receives config through an IPC mechanism designed to avoid blocking the audio server.
4. Driver atomically swaps the active graph between render cycles.

## Shared Configuration ABI

`Sources/HALDriverC/include/HeartechoHALShared.h` now defines the fixed C ABI for the compact runtime configuration. The Swift bridge mirrors this layout in `HALDriverBridge.sharedConfigurationData(from:)` and default diagnostics decode the bytes again to verify the header and first device. The SwiftUI app writes this snapshot next to `RoutingGraph.json` as `HALSharedConfig.bin` and publishes it to POSIX shared memory whenever the routing store saves.

The current ABI contains:

- 32-byte header with magic, version, struct sizes, device count, and maximum device count.
- 16 fixed-size device slots.
- 256 bytes per device: device/input stream/output stream object IDs, channel count, sample rate, enabled flag, latency frames, safety offset frames, buffer frame size, UTF-8 name, and UTF-8 UID.
- Stable object numbering: device object IDs start at 2 and use a stride of 3.

The C driver now has non-real-time file and shared-memory loaders for this ABI. `HeartechoHALDriverLoadSharedConfigFromFile` reads a complete fixed-size file snapshot, and `HeartechoHALDriverLoadSharedConfigFromSharedMemory` copies a fixed-size POSIX shared-memory snapshot. Both paths validate the header and atomically switch the active in-process config pointer. Default diagnostics write the Swift-generated bytes to a temporary file, publish them to a temporary shared-memory object, load both through the C API, and verify the active device name, UID, object ID, sample rate, channel count, latency, safety offset, and buffer frame size.

When a new config is activated, the C driver compares the previous and next snapshots, records a compact `HeartechoHALConfigChangeSummary`, and builds Core Audio `PropertiesChanged` calls for plug-in device-list changes, device name/UID changes, sample-rate or channel-count changes, and stream format changes. Diagnostics validate this model in-process without needing the driver to be installed.

The C driver also answers common runtime device properties used by Core Audio clients: configured latency, configured safety offset, configured buffer size bounded by the shared audio-buffer capacity, and `kAudioDeviceTransportTypeVirtual`. Diagnostics query these through the driver property path directly, so the behavior is covered without installing the HAL bundle.

The next production step is to put a signed helper in charge of publishing the shared-memory snapshot and to add the surrounding request/commit flow for structural device changes in a live AudioServerPlugIn. The render callback must continue to avoid file I/O and blocking synchronization.

`HeartechoHelper` is the current command-line scaffold for that helper boundary. It loads `RoutingGraph.json`, can create a starter graph for development with `--init-starter-graph`, publishes the HAL config shared-memory snapshot, and can open/write live audio shared memory with `--publish-audio`. It also supports `--serve` with `--interval-ms` and optional `--iterations`, so development and diagnostics can exercise a finite or continuous publication loop before it becomes a signed launchd helper.

`scripts/build-helper-launch-agent.sh` generates and lints a user LaunchAgent plist for this helper, while `scripts/install-helper-launch-agent.sh` and `scripts/uninstall-helper-launch-agent.sh` preview copy/load and unload/remove operations by default. `HelperServiceProbe` reports the built plist plus user and system LaunchAgent install locations to diagnostics and the app readiness panel, matching the product package payload under `/Library/LaunchAgents`. Real production use still requires a signed helper binary, an approved launchd load path, and lifecycle recovery when the helper or audio server restarts.

## Audio Buffer ABI

`Sources/HALDriverC/include/HeartechoHALAudioShared.h` defines the current audio data shape: up to 16 fixed 64-channel, 4096-frame float buffer slots with counters for written, read, available, and dropped frames, wrapped in a fixed-size audio shared state. `HALAudioBufferBridge` can open this state as a live POSIX shared-memory mapping, interleave Swift `MixedAudioBuffer` values into the C API, and close/unlink the mapping for diagnostics or lifecycle cleanup. `HALRenderPublisher` maps `RuntimeRenderReport` device IDs to the object IDs from the HAL shared configuration before writing rendered frames.

The C `DoIOOperation` path now tries to read frames from the buffer slot for the requested device object and zero-fills any missing frames. On driver initialization, the C driver attempts to load the default config shared memory and open the default live audio shared-memory object, so an installed driver can attach to helper-published state without test-only calls. Default diagnostics verify this without installing the driver by writing rendered mixes through Swift, reading them back through the C API, checking a two-device publication case, publishing a snapshot-compatible audio state, and checking a close/reopen live shared-memory transport case.

The C driver now exposes `HeartechoHALRealtimeSafetyStats` for diagnostics. The stats count `DoIOOperation` audio reads, frames read, zero-filled frames, and tracked render-path risks. Diagnostics drive the C `DoIOOperation` read path directly and require lock, allocation, file-I/O, and shared-memory-open counters to stay at zero. This is an in-process audit of the render primitive; production still needs the same invariant validated after a signed installed driver is called by Core Audio over a sustained run.

Source buffers can now carry capture sample-rate metadata. `RuntimeRoutingEngine` converts mismatched source buffers to the target virtual-device sample rate before mixing and publishing, so the HAL data plane receives frames in the configured device clock domain. The default converter is a balanced windowed-sinc path, with explicit linear fallback for low-cost diagnostics and a wider mastering mode available for future user-facing quality controls. Live/external source buffers also pass through a conservative buffer-watermark drift controller that nudges the resampling ratio when the capture buffer is under or over its target waterline, and the runtime keeps per-device/per-source resampling phase state across render blocks. Production still needs tighter clock recovery and long-run SRC/drift validation.

Default diagnostics also verify `HeartechoHelperRuntime` by saving a temporary graph, publishing config/audio shared memory through the helper runtime, loading the config through the C HAL shared-memory API, reopening the audio state through the C audio shared-memory API, checking the resulting device count and buffer counters, and running a finite helper loop to confirm repeated publication.

This is not the final production data plane. The current live shared-memory path removes the earlier mutex-protected read path and gives helper-owned writes a cross-process buffer, but it still needs signed installation, launchd ownership, lifecycle recovery, and a stricter real-time safety review before it should be treated as production.

## Process Tap Capture

The app now discovers running regular applications through `NSWorkspace` for the visible app menu and also keeps a background capture-candidate list for Loopback-style special sources. Special sources are stored with stable `special:` identifiers and expand to one or more bundle/process identifiers before a process tap is created.

1. Translate selected bundle identifiers or process identifiers into Core Audio process objects. Current status: implemented for PID diagnostics and multi-process special-source configuration.
2. Create a process tap for those objects. Current status: implemented by `CoreAudioProcessTapManager.createStereoMixdownTap`.
3. Feed tap output into a source ring buffer keyed by `AudioSource.id`. Current status: `ProcessTapCaptureSession` creates a private aggregate device, installs an IOProc, and writes captured float samples to `AudioRingBuffer`.
4. Call the same route evaluation semantics used by `HeartechoAudio.RoutingMixer`.
5. Keep permission failures and ended processes visible in the app source cards.

`ProcessTapCaptureSession.prepare()` creates the tap/aggregate/IOProc without starting capture. `start()` begins IO and should only be called for an explicitly selected source after permissions and UI state are clear. Because aggregate/tap creation can block depending on current HAL and permission state, default diagnostics do not call `prepare()`; use `swift run HeartechoDiagnostics --prepare-process-tap` for an explicit probe.

This is separate from publishing virtual devices. Process taps provide app audio; the HAL plug-in publishes the device other apps can select.

## Hardware Input Capture

Hardware input sources now have a parallel capture path through `HardwareInputCaptureSession`. The session resolves a selected Core Audio input or duplex device, creates an `AudioDeviceIOProcID`, ingests float input buffers into `AudioRingBuffer`, and exposes the same capture state model used by application sources. Source identifiers prefer Core Audio device UIDs and fall back to object IDs for existing saved graphs.

Default diagnostics do not prepare or start hardware input capture because it can interact with microphone/device permissions. Use `swift run HeartechoDiagnostics --prepare-hardware-input` for an explicit non-starting IOProc creation probe.

## Current Repository State

`Sources/HALDriverStub` is intentionally not a complete HAL driver yet. It defines the Swift-side validation bridge, fixed shared config encoder, shared-memory publisher, and exported factory symbol placeholder so the target boundary exists early. `Sources/HALDriverC` contains the compiling C AudioServerPlugIn skeleton, shared-config loaders, change-notification model, prototype audio buffer, and device metadata queries backed by the active config snapshot.

`HeartechoAudio.RoutingMixer` is the current non-real-time reference for route evaluation. `HeartechoAudio.HALRenderPublisher` is the current diagnostic publisher from that rendered output into the C buffer and prototype audio shared-memory state. The production driver render path should match this behavior, then replace arrays and the prototype mutex snapshot with real-time safe storage.

`Sources/HALDriverC` now contains the C AudioServerPlugIn skeleton, `HALBundle/Heartecho.driver` contains the bundle template, and `scripts/build-hal-bundle.sh` can create an unsigned local bundle artifact. See `docs/hal-bundle.md`.
