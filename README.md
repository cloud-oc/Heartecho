# Heartecho

Heartecho is a native macOS Swift application for building a lightweight audio routing tool inspired by Rogue Amoeba Loopback.

The project is intentionally VSCode friendly:

- Swift Package Manager layout
- SwiftUI/AppKit desktop UI
- Local `.app` bundle build and verification scripts for macOS packaging previews
- Native Settings window and toolbar appearance selector for Match System, Light, and Dark UI themes
- Generated app and HAL driver icon assets, with bundle metadata and verification coverage
- Installer `.pkg` staging/build/verification scripts for app, HAL driver, helper, and LaunchAgent payloads
- Script-only uninstaller `.pkg` build/verification scripts for clean app, HAL driver, helper, and LaunchAgent removal
- Distribution product `.pkg` build/verification scripts that wrap installer and uninstaller component packages
- Product signing audit and notarization dry-run scripts for release preflight
- JSON release manifest generation with artifact hashes, bundle metadata, signing state, notarization staple checks, and release gates
- Installer lifecycle scripts for preinstall cleanup, helper bootstrap, and Core Audio reload requests
- Post-install validation script for app, HAL driver, helper, LaunchAgent, launchd state, and Core Audio virtual-device visibility
- Installed-driver audio validation loop for Core Audio visibility, readiness, and live HAL audio transport checks
- Core Audio device discovery services
- Running application source discovery
- Codable routing model for virtual devices, application sources, special system application sources, hardware sources, nested virtual-device sources, channel maps, monitors, and pass-thru paths
- Inline native name editing for virtual devices, sources, and monitors, with virtual device names reflected in HAL shared configuration
- Virtual device enable/disable, deletion, sample-rate, buffer-frame-size, latency, and safety-offset controls, reflected in HAL shared configuration and active-device filtering
- Editable source-to-output routing with up to 64 output channels
- Device master gain/mute plus per-source mute/gain, per-route mute/gain, and per-monitor mute/gain controls
- Offline routing mixer used as the reference implementation for graph semantics
- Runtime routing engine that pulls source buffers/capture sessions, resolves nested virtual-device sources, and renders virtual device output
- Runtime sample-rate conversion for source buffers whose capture clock differs from the target virtual device sample rate, using a default windowed-sinc converter with linear fallback
- Conservative buffer-watermark drift correction with continuous resampling phase for live source render blocks
- Source, route, and virtual output-channel meters driven by the latest runtime render report
- Monitor output engine that receives virtual device mixes and tracks monitor buffer/peak state
- Hardware monitor playback sessions using AudioQueue for selected output-device UIDs and AVAudioEngine for default-output fallback
- Monitor output device selection using Core Audio device UIDs, with monitor enable, mute, gain, channel routing, playback, and remove controls
- File-based JSON preset import/export for routing graphs
- Local preset library for saving, applying, importing, deleting, tagging, searching, filtering, and migrating routing graph presets
- Audio ring buffer and level meter primitives for source capture paths plus source, route, and virtual output metering
- Core Audio process tap capability and process-object diagnostics
- OS-aware special application source presets for Finder, Siri, Sound Effects, VoiceOver, Background Sounds, Notification Center, Spoken Content, and System AirPlay Receiver, including stable `special:` identifiers and captured-process identifier sets for multi-process system sources
- Process tap aggregate capture session scaffold that can feed app audio into source ring buffers, including background capture-candidate discovery and multi-process tap configuration for special system sources
- Application sources can opt into process-tap mute-when-capturing behavior while leaving existing captures unmuted by default
- Hardware input capture session scaffold that can feed Core Audio input devices into source ring buffers
- App-side application and hardware input capture controls with prepare/start/stop/release state
- App-side readiness and permissions panel for HAL driver state, virtual-device visibility, system audio/process-tap access, microphone permission, input/output devices, and live HAL audio transport
- App bundle privacy usage descriptions for microphone input and macOS system/application audio capture
- Compiling C HAL Audio Server Driver skeleton and `.driver` bundle template
- Fixed HAL shared configuration ABI generated by Swift and consumed by the C driver skeleton
- App-side persistence writes `RoutingGraph.json`, writes `HALSharedConfig.bin`, and publishes a POSIX shared-memory config snapshot
- `HeartechoHelper` executable that can bootstrap a starter graph and publish HAL config/live audio shared-memory state
- Helper signing plan, LaunchAgent generation, install/uninstall dry-run scripts, and app/diagnostic probing for built, user-installed, or system-installed helper service plists
- Helper service recovery dry-run planning for launchd kickstart/restart after stale transport detection
- Product-level install/uninstall workflow dry-run scripts that orchestrate HAL bundle, helper service, Core Audio reload, and visibility checks
- C HAL file and shared-memory config loaders verified by diagnostics without installing a driver
- C HAL config-change summary and property notification skeleton for device metadata/format updates
- C HAL runtime device properties for latency, safety offset, buffer frame size/range, and virtual transport type
- Multi-device HAL audio buffer ABI and live POSIX shared-memory transport for rendered runtime mixes
- HAL audio transport writer/reader heartbeat counters with stale writer and overflow readiness diagnostics
- C HAL realtime safety stats for the `DoIOOperation` audio read path

## Requirements

- macOS 14 or newer
- Swift 6 toolchain
- Full Xcode is recommended for signing and installing a HAL plug-in, although the current app scaffold builds with Command Line Tools.

## Run

```sh
swift run Heartecho
```

In VSCode, use `Terminal > Run Task > Swift: Run App`.

To build and verify a local `.app` bundle without opening it:

```sh
scripts/build-icons.sh
scripts/build-app-bundle.sh debug
scripts/verify-app-bundle.sh
```

`scripts/build-icons.sh` generates complete app and HAL driver `.iconset` resources and attempts to build `.icns` files with `iconutil`. In this Codex sandbox, `iconutil` may reject generated iconsets because macOS provenance metadata cannot be removed; bundle verifiers therefore accept either `.icns` or the complete iconset PNG resource set.

To exercise the helper publication scaffold without installing a HAL driver:

```sh
swift run HeartechoHelper --init-starter-graph --publish-audio
```

For a finite preview of the service loop that keeps HAL config/audio shared memory refreshed:

```sh
swift run HeartechoHelper --init-starter-graph --publish-audio --serve --iterations 3 --interval-ms 10
```

To generate and lint a non-installed LaunchAgent plist for the helper service:

```sh
scripts/build-helper-launch-agent.sh
scripts/install-helper-launch-agent.sh
scripts/uninstall-helper-launch-agent.sh
scripts/recover-helper-service.sh --kickstart
```

To preview the product-level install or uninstall workflow without copying files, loading launchd, or restarting Core Audio:

```sh
scripts/install-heartecho.sh --wait 30
scripts/uninstall-heartecho.sh --unload-helper --reload-core-audio
```

Add `--execute` only after signing/notarization and install-location decisions are ready.

To stage and build a local installer package artifact without running the installer:

```sh
scripts/build-installer-pkg.sh --execute
scripts/verify-installer-pkg.sh
scripts/build-uninstaller-pkg.sh --execute
scripts/verify-uninstaller-pkg.sh
```

The verifier also checks installer lifecycle scripts. On some local macOS volumes, `pkgbuild` may preserve provenance metadata as AppleDouble sidecar entries; the verifier reports that as a warning while still requiring the primary payload and scripts.

To wrap the installer and uninstaller components in a distribution product package:

```sh
scripts/build-distribution-product.sh --execute
scripts/verify-distribution-product.sh
scripts/check-product-signing.sh
scripts/notarize-product-pkg.sh --keychain-profile ExampleProfile
```

The product package workflow is also dry-run by default unless `--execute` is supplied. `scripts/notarize-product-pkg.sh --execute` requires a signed Developer ID Installer package plus Apple notary credentials.

For a no-system-changes release preflight that builds the app/HAL/helper/package artifacts, runs sandbox-safe diagnostics, verifies package structure, previews helper recovery, checks VSCode task JSON, and writes a consolidated report:

```sh
scripts/release-preflight.sh
```

The report is written to `build/release-preflight-report.txt`, and the artifact manifest is written to `build/release-manifest.json`. The manifest records each release artifact path, size, file count, SHA-256 digest, bundle plist metadata, signing state, notarization staple validation, and release gate status. Unsigned or ad-hoc artifacts are reported but do not fail the default preflight; use `scripts/release-preflight.sh --require-signed` after Developer ID Application signing for the app/HAL/helper and Developer ID Installer signing for the packages are in place. In VSCode, use `Terminal > Run Task > Product: Release Preflight`.

To refresh just the manifest after signing or notarization work:

```sh
scripts/write-release-manifest.sh
scripts/write-release-manifest.sh --require-signed
```

The `--require-signed` form intentionally fails until Developer ID signatures and stapled notarization tickets are present.

To audit a real installation without modifying the system:

```sh
scripts/validate-installation.sh --wait 30
```

Add `--strict` after installing when failures should make the command fail.

To run the installed-driver audio validation loop without modifying the system:

```sh
scripts/validate-installed-audio.sh --iterations 3 --wait 30
scripts/validate-installed-audio.sh --strict --iterations 3 --wait 30
```

This report is written to `build/installed-audio-validation-report.txt`. It checks installed payloads, helper launchd state, Core Audio virtual-device visibility, readiness state, and repeated diagnostics. Use `--skip-shared-memory` only in restricted environments; production validation should include the live shared-memory transport checks.

In VSCode, use `Terminal > Run Task > Swift: Run Helper`, `Helper: Build LaunchAgent Plist`, `Helper: Install LaunchAgent Dry Run`, or `Helper: Uninstall LaunchAgent Dry Run`.

To build and verify a local HAL `.driver` bundle artifact without installing it:

```sh
scripts/build-hal-bundle.sh debug
scripts/verify-hal-bundle.sh
```

In VSCode, use `Terminal > Run Task > HAL: Build Bundle`, then `HAL: Verify Bundle`.

To inspect signing and preview install/uninstall actions:

```sh
scripts/check-hal-signing.sh
scripts/sign-hal-bundle.sh --identity "Developer ID Application: Example"
scripts/notarize-hal-bundle.sh --keychain-profile ExampleProfile
scripts/install-hal-bundle.sh
scripts/uninstall-hal-bundle.sh
```

The sign/notarize/install/uninstall scripts are dry-run by default. Install requires a valid signature when run with `--execute`; notarization execute mode also requires real Apple notary credentials. Use `--allow-unsigned` only for temporary development installs.

After a signed install, preview Core Audio reload and wait for the virtual device:

```sh
scripts/reload-core-audio.sh
swift run HeartechoDiagnostics --wait-hal-device 30
```

The reload script is dry-run by default; add `--execute` only when you explicitly want macOS to restart `coreaudiod`.

## Diagnostics

```sh
swift run HeartechoDiagnostics
```

The diagnostics target performs a quick routing-graph validation and Core Audio device enumeration without needing XCTest. Full `swift test` support requires a complete Xcode installation on this machine.

For CI or Codex sandbox runs where POSIX shared memory is blocked, use:

```sh
CLANG_MODULE_CACHE_PATH=.build/clang-module-cache swift run --disable-sandbox HeartechoDiagnostics --skip-shared-memory
```

The `--disable-sandbox` flag disables SwiftPM's nested sandbox only; Codex or your shell environment may still enforce its own filesystem rules. This mode still checks routing, HAL config encoding, C HAL file loading, device property behavior, mixer/render/monitor/meter models, capture state models, app/device discovery, and packaging probes. It explicitly skips C HAL shared-memory loading, helper shared-memory publication, live HAL audio shared-memory publication, and the readiness audio-transport publish check, so it is not a replacement for full installed-driver validation.

To explicitly attempt creating a private process-tap aggregate capture session, run:

```sh
swift run HeartechoDiagnostics --prepare-process-tap
```

That command can block or prompt depending on the current macOS audio permission and HAL state, so the default diagnostics path skips it.

To explicitly attempt creating a hardware input IOProc without starting capture, run:

```sh
swift run HeartechoDiagnostics --prepare-hardware-input
```

That command can interact with microphone/input-device permission state, so the default diagnostics path skips it.

To explicitly attempt starting and immediately stopping monitor hardware playback, run:

```sh
swift run HeartechoDiagnostics --start-monitor-playback
```

This can produce audio on the selected/default output device, so the default diagnostics path skips it.

To run a longer HAL audio transport stress diagnostic, use:

```sh
swift run HeartechoDiagnostics --stress-hal-transport
```

The default sandbox-safe diagnostics already include an in-memory multi-iteration HAL transport stress check for writer/reader heartbeats, read/write counters, overflow detection, and realtime-read counters. The `--stress-hal-transport` option also attempts the POSIX shared-memory transport loop, so it should be run in a local environment that can create POSIX shared-memory objects and without `--skip-shared-memory`.

The current diagnostics check:

- HAL runtime configuration generation

- buffer-watermark drift correction and continuous resampling phase behavior for resampled live sources
- helper publication runtime for config and live audio shared memory, unless `--skip-shared-memory` is supplied
- HAL audio buffer bridge, runtime render publisher, multi-device render publication, and live audio shared-memory transport, with POSIX shared-memory transport skipped by `--skip-shared-memory`
- multi-iteration HAL audio transport stress for writer/reader heartbeat movement, read/write counters, overflow detection, and realtime-read counters

## Reality Check

A Loopback-equivalent product cannot be finished as a plain sandboxed app. The user-facing app can be SwiftUI, but the virtual devices that appear in macOS sound settings require a Core Audio HAL Audio Server Driver plug-in installed in the correct system or user plug-in location. Capturing per-application audio requires macOS process taps, an `NSAudioCaptureUsageDescription` purpose string, and user-granted audio permissions; hardware input capture requires `NSMicrophoneUsageDescription` and microphone permission. The app surfaces those prerequisites in the Inspector readiness panel, including a manual microphone-access request button. Diagnostics only read permission state by default and do not trigger privacy prompts.

This repository starts both sides:

- `HeartechoApp`: native configuration and monitoring app
- `HeartechoCore`: product model and routing graph
- `HeartechoAudio`: Core Audio discovery, running application discovery, mixer/runtime render pipeline, monitor playback, and prototype HAL render/shared-memory publication
- `HeartechoHelper`: command-line helper scaffold for graph bootstrap plus HAL config/audio shared-memory publication
- `HALDriverStub`: Swift-side HAL validation/configuration bridge and exported symbols for the HAL plug-in path
- `HALDriverC`: compiling C AudioServerPlugIn skeleton with shared configuration, audio shared-memory state, and prototype audio-buffer APIs

See [docs/architecture.md](docs/architecture.md) and [docs/implementation-plan.md](docs/implementation-plan.md).
